# MiniOpsAgent 实现说明

本文档从代码角度解释 MiniOpsAgent 的关键实现。

## 1. CLI 入口

入口文件：

```text
cmd/miniopsagent/main.go
```

核心函数：

- `main`：创建带系统信号的 context，执行 root command。
- `rootCommand`：注册根命令、flag 和子命令。
- `bootstrap`：组装 Config、LLM、Skill、RAG、Snapshot、Tools、MCP 和 Agent。
- `runPlain`：普通文本 REPL。
- `runAgentInput`：根据 mode 或 `/plan`、`/team` 前缀分发。

`bootstrap` 是理解项目的第一入口，因为所有核心依赖都在这里连接起来。

## 2. 配置加载

文件：

```text
internal/config/config.go
```

配置加载顺序：

1. `defaults()` 创建默认 provider 配置。
2. 读取 `.env` 和 `~/.env`。
3. 读取 `~/.miniopsagent/config.json`。
4. `fillProviderDefaults` 补齐缺失字段。
5. `applyEnv` 使用环境变量覆盖配置。
6. `applyModelDefaults` 根据模型名修正上下文长度。

重要环境变量：

```text
MINIOPS_PROVIDER
MINIOPS_API_KEY
MINIOPS_BASE_URL
MINIOPS_MODEL
```

`MINIOPS_*` 作用于当前默认 provider；`DEEPSEEK_API_KEY`、`OPENAI_API_KEY` 等作用于指定 provider。

## 3. LLM 客户端

文件：

```text
internal/llm/client.go
internal/llm/types.go
```

MiniOpsAgent 使用 OpenAI-compatible Chat Completions 协议。

普通请求：

```text
Client.Chat
  -> chatRequest(stream=false)
  -> 解析 choices[0].message
  -> 返回 content / reasoning_content / tool_calls
```

流式请求：

```text
Client.ChatStream
  -> chatRequest(stream=true)
  -> 逐行读取 SSE data
  -> 累积 content
  -> 累积 reasoning
  -> 按 index 拼接 tool_calls
```

工具调用参数兼容两种形式：

```json
{"path":"README.md"}
```

以及被模型编码成字符串的形式：

```json
"{\"path\":\"README.md\"}"
```

`normalizeFunctionArguments` 会把第二种转回 JSON object。

## 4. Agent 主循环

文件：

```text
internal/agent/agent.go
```

核心函数：

```text
RunWithObserver
```

流程：

1. 清理空输入。
2. 创建 pre-turn 快照。
3. 刷新系统提示。
4. 追加用户消息。
5. 调用 LLM。
6. 如果没有 tool call，直接返回最终回答。
7. 如果有 tool call，执行工具并把结果追加到历史。
8. 最多循环 10 次。
9. 如果重复调用同一个工具，触发 loop guard。
10. 创建 post-turn 快照。

系统提示包含：

- 当前日期。
- provider 和 model。
- Skill 索引。
- 相关长期记忆。
- 工具使用策略。

## 5. 运行模式

### ReAct

默认模式。模型决定是否调用工具。

### Plan

`PlanAndExecuteWithObserver` 先让模型生成计划，再把计划和原始需求交给 ReAct 执行。

### Team

`Team` 使用 Planner、Worker、Reviewer 三个角色串行处理输入。它更像教学示例，展示多角色工作流，而不是生产级多 Agent 框架。

## 6. 工具注册和执行

文件：

```text
internal/tools/registry.go
```

注册入口：

```text
registerBuiltins
```

执行入口：

```text
ExecuteAll
Execute
```

`ExecuteAll` 会并发执行多个工具调用，最多同时 4 个。每个工具调用有 90 秒超时。

`Execute` 会：

1. 解析 JSON 参数。
2. 查找工具。
3. 执行工具。
4. 截获错误并转成工具结果。
5. 对危险工具和 MCP 工具写审计日志。

## 7. 文件工具

文件工具包括：

```text
read_file
write_file
list_dir
glob_files
grep_code
create_project
```

所有文件路径先经过 `PathGuard.Resolve`，确保位于工作区内。

`write_file` 单次写入限制 5 MB，并在写 Go 文件后触发 `GoDiagnosticsHook`。该 hook 会尝试执行：

```bash
go test ./...
```

如果本机没有 Go，会返回诊断错误。

## 8. 命令工具

工具：

```text
execute_command
```

命令执行前经过 `CommandGuard`。

执行 shell：

- Windows：`powershell.exe -NoProfile -NonInteractive -ExecutionPolicy Bypass -Command`
- Linux/macOS：`sh -lc`

输出最多返回 12000 字符，避免超大日志塞爆上下文。

## 9. Web 工具

文件：

```text
internal/tools/web.go
```

搜索优先级：

1. SearXNG。
2. SerpAPI。
3. DuckDuckGo HTML fallback。

网页抓取：

```text
web_fetch
  -> url.Parse
  -> scheme 检查
  -> validatePublicHost
  -> httpGet
  -> extractHTMLText
```

`validatePublicHost` 会解析域名并拒绝非公网 IP，降低 SSRF 风险。

## 10. MCP 实现

文件：

```text
internal/tools/mcp.go
```

MCP 配置来源：

```text
~/.miniopsagent/mcp.json
.miniopsagent/mcp.json
```

支持两类 server：

- stdio：启动本地命令，通过 stdin/stdout JSON-RPC 通信。
- HTTP：向远端 URL 发送 JSON-RPC POST。

启动后会调用：

```text
initialize
tools/list
resources/list
```

远端工具会注册成本地工具：

```text
mcp__server__tool
```

## 11. RAG 实现

文件：

```text
internal/rag/index.go
```

当前 RAG 是轻量本地实现，不依赖向量数据库。

Build 流程：

1. 遍历目录。
2. 跳过 `.git`、`.miniopsagent`、`node_modules`、`dist` 等目录。
3. 对代码和文档文件建立 file chunk。
4. 对 Go 文件额外解析函数符号。
5. 记录 import 和 contains 关系。

Search 流程：

1. 对 query 分词。
2. 和 chunk terms 做余弦相似度。
3. path 命中加分。
4. symbol 命中加分。
5. 返回 topK。

这是教学友好的实现，后续可以替换为 embedding。

## 12. Skill 实现

文件：

```text
internal/skill/skill.go
```

Skill 来源：

```text
builtin
~/.miniopsagent/skills
.miniopsagent/skills
```

每个 Skill 是一个目录，包含：

```text
SKILL.md
```

`parseFrontmatter` 会解析 name、description、version、author、tags。系统提示里只注入 Skill 索引，完整内容通过 `load_skill` 延迟注入下一轮用户消息。

## 13. Snapshot 实现

文件：

```text
internal/snapshot/snapshot.go
```

快照目录按工作区绝对路径 hash 隔离：

```text
~/.miniopsagent/snapshots/<workspace-hash>/<snapshot-id>/
```

创建快照时会复制工作区，跳过：

```text
.git
.miniopsagent
node_modules
target
dist
build
coverage
vendor
```

单个超过 20 MB 的文件不会复制。

## 14. Runtime API 实现

文件：

```text
internal/runtime/server.go
```

当前实现是本地内存 runtime。

核心结构：

```text
Server
Thread
Event
```

请求流：

```text
POST /v1/threads
  -> 创建 Thread

POST /v1/threads/{id}/turns
  -> 记录 turn.started
  -> goroutine 执行 Agent
  -> 记录 message.delta
  -> 记录 turn.completed / turn.failed

GET /v1/threads/{id}/events
  -> 输出已有事件
```

认证使用 Bearer token 或 `X-MiniOpsAgent-API-Key`。

## 15. TUI 实现

文件：

```text
internal/tui/tui.go
```

TUI 基于 Bubble Tea：

- `model` 保存界面状态。
- `Update` 处理键盘、鼠标、窗口尺寸和流式事件。
- `View` 渲染 banner、transcript、input 和 status bar。
- `runPrompt` 在 goroutine 中调用 Agent。
- `applyStreamEvent` 把 thinking、tool call、tool result 和 answer delta 映射为界面输出。

TUI 测试覆盖了滚动、输入框布局、状态栏宽度、终端控制序列清理和流式输出合并。
